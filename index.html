GenericAgent 拆解课
01

你扔一句话，它接管整台电脑

认识 GenericAgent：一个只有约 3000 行代码，却能自己开浏览器、敲命令、点手机的智能体。

先看一个真实场景

你打开一个聊天框，敲下一句话：「帮我点一杯奶茶」，回车。

几十秒后，外卖软件被自己操作着选好了门店、加好了购物车、走到了结算页。你全程没碰鼠标。

🤖
这就是「智能体」（Agent）

一个 LLM 大模型 负责「思考」，外面包一层程序负责「动手」——读屏幕、敲键盘、点网页。两者合起来，就能把一句人话变成一连串真实操作。

为什么它值得拆开看？

市面上的智能体框架动辄几十万行代码。GenericAgent 反着来：用极少的「种子代码」，让能力自己长出来。

🪶

约 3000 行

核心代码极小。对比 OpenClaw 的 ~53 万行，它几乎是一张草稿纸的体量。

🧰

9 件原子工具

跑代码、读写文件、看网页……就这 9 件最基础的「手」，组合出一切能力。

🔁

约 100 行的循环

整个「思考→行动→再思考」的引擎，只有一百行左右。下一章我们逐行读它。

🧬

越用越强

每解决一个新任务，它把过程「结晶」成一个技能存进记忆，下次直接复用。

💡
设计哲学：不预装技能，而是让它进化

这正是这门课的主线。代码这么少，秘密全在「怎么组织」，而不是「写了多少」。

关键直觉

智能体不是「一次性想好全部步骤」，而是走一步看一步。这个不停打转的循环，就是整个框架的心脏——也是下一章的主角。

这趟旅程从哪一行代码开始

所有这些，都从一个叫 agent_runner_loop 的函数起步。它做的第一件事，就是把「系统说明书」和「你的话」摆上桌。

CODE · agent_loop.py 
def agent_runner_loop(client, system_prompt, user_input, handler, tools_schema,
                      max_turns=40, verbose=True, ...):
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": ... user_input}
    ]
大白话

定义那个核心循环函数。它收到几样东西：一个能和大脑对话的 通话客户端、一份系统说明书、你的那句话、一个会真正动手的 执行器，和一份工具清单。

max_turns=40 是一道保险：最多自己转 40 圈就停，免得无限打转。

开一个对话列表 messages，这就是要发给大脑的内容。

第一条是 系统说明书，告诉大脑「你是谁、能用什么工具、规矩是什么」。

第二条才是你说的那句话。

说明书 + 你的话，凑成大脑的第一份「输入」。旅程正式开始。

02

100 行的心跳

把整个智能体引擎想成一局「回合制游戏」——每一回合，AI 只做一件事。

一个好用的比喻：回合制游戏

象棋、回合制 RPG，都是「轮到你 → 你走一步 → 看棋盘变化 → 再轮到你」。GenericAgent 的引擎一模一样。

🧠
轮到 AI 行动

把当前对话发给大脑，让它想：「这一步我要调用哪个工具？」

🦾
执行这一步

引擎把大脑点名的工具真正跑一遍，拿到结果。

🔄
把棋盘变化告诉它

结果被打包成下一回合的输入，再丢回大脑。回合数 +1。

⏱️
一个「回合」= 一次工具调用

工程师把这一圈叫做一个 turn（回合）。下面的代码里，turn 这个计数器就是游戏的回合数。

心跳的第一拍：开口问大脑

每个回合的开头，引擎做的第一件事，是把整份对话发给大脑、等它回话。

CODE · agent_loop.py 
while turn < handler.max_turns:
    turn += 1
    response_gen = client.chat(messages=messages, tools=tools_schema)
    response = yield from response_gen
大白话

只要回合数还没到上限，就继续转这个圈。

回合数加 1——新的一回合开始了。

把整份对话 + 可用工具清单，发给大脑，请它思考并回话。

等大脑把话说完，收下它的完整回复 response。

📡
为什么把工具清单一起发？

大脑不会凭空知道自己有哪些「手」。每回合都把 tools_schema（工具说明书）一起递过去，它才知道这回合能选哪些工具来用。

第二拍：大脑点名，引擎动手

大脑的回话里，藏着它这一步想调用的工具。引擎要把这些「点名」一个个解析出来，然后真正去执行。

CODE · agent_loop.py 
tool_calls = [{'tool_name': tc.function.name,
               'args': json.loads(tc.function.arguments),
               'id': tc.id}
              for tc in response.tool_calls]
大白话

从大脑的回复里，把每一个「工具调用」拆出来。

记下工具的名字（比如 code_run）。

记下要传给它的参数。JSON 文本在这里被翻译回程序能用的形式。

还有一个 id，方便稍后把「结果」对回到「是哪次调用」。

大脑这回合可能一次点好几个工具，所以用列表把它们都收齐。

把一个回合，听成一段对话

如果给引擎、大脑、工具三个角色配上对话框，一个回合大概是这样聊的：

⚙️
引擎 (Loop)

第 7 回合。这是目前的对话和你能用的工具，想想下一步该干嘛？

🧠
大脑 (LLM)

我决定调用 code_run，跑这段 Python 把价格抓下来。

⚙️
引擎 (Loop)

收到，我去找名叫 do_code_run 的执行器，把它跑了。

🦾
工具 (code_run)

跑完了，结果：{"price": 18}。给你一个 StepOutcome。

⚙️
引擎 (Loop)

它没说「该退出」，那就把结果打包成下一回合的输入。第 8 回合，开始！

?

第三拍：这回合的「裁决」

每个工具跑完，都会交回一张小纸条，叫 StepOutcome。引擎只看上面三栏，就知道这局怎么继续。

CODE · agent_loop.py 
@dataclass
class StepOutcome:
    data: Any
    next_prompt: Optional[str] = None
    should_exit: bool = False
大白话

定义一张标准化的「结果纸条」。每个工具跑完都填一张。

data：这一步实际拿到的东西（比如抓到的价格）。

next_prompt：下一回合要对大脑说的话。如果留空，引擎就认为「任务做完了」，停。

should_exit：一个 真假开关。比如工具要找用户确认，就把它设成「真」，主动跳出循环等人。

🧭
三栏纸条，就是整局游戏的方向盘

「有没有下一句话」决定继续还是收工；「要不要退出」决定是否暂停等人。一个极简的约定，控制了整个循环的走向——这就是少代码却不乱的诀窍。

心跳的最后一拍：只带「新消息」进下一回合

每回合结束，引擎都会做一件乍看很怪的事——把对话列表清空，只留下这一条新消息。

CODE · agent_loop.py 
# just new message, history is kept in *Session
messages = [{"role": "user",
             "content": next_prompt,
             "tool_results": tool_results}]
大白话

代码作者亲手写的注释：「只放新消息，历史另外存着」。

把 messages 整个重置，里头只剩一条新的「用户消息」。

内容是上一步的裁决（next_prompt）……

外加这回合工具跑出来的结果。然后下一回合就从这里继续。

🤔
等等——那之前的对话不就丢了吗？

这正是 GenericAgent 最聪明、也最反直觉的设计，直接关系到它「省 token、少出错」的核心优势。我们专门留到 第 4 章 拆它。先记住这个疑问。

03

9 件原子工具

就像乐高只有几种基础颗粒，却能拼出整座城堡——GA 只用 9 件最基础的「手」，组合出一切能力。

「原子」是什么意思

原子，就是不能再拆小的最基础零件。GA 的工具刻意保持极少、极基础，复杂能力靠它们组合出来，而不是一个个写死。

🧱
乐高思维

乐高不会为「城堡」「飞机」各做一种专用零件，而是给你几种通用颗粒，让你自己拼。GA 的 9 件工具就是这几种颗粒——原子工具。

认识这 9 件颗粒

它们分三类：动手干活的、读写文件的、还有两件特殊的「记忆」工具。

⚡

code_run

执行任意代码（Python / PowerShell）。最万能的一件，下一屏专门讲它。

📖

file_read

读取文件内容，可从指定行开始、可按关键词定位。

✍️

file_write

整体写入 / 新建 / 覆盖一个文件，适合大段内容。

🩹

file_patch

只改文件里的一小段，做精细修改而不动其余部分。

🌐

web_scan

感知当前网页内容、列出标签页，也能切换标签页。

🕹️

web_execute_js

在真实浏览器里跑任意 JS，对网页实现「完全控制」。

🙋

ask_user

向人求助、确认关键决策。这是「人在回路」的安全阀。

📝

update_working_checkpoint

（记忆）短期的「便利贴」，记下当前任务的重点。

💎

start_long_term_update

（记忆）把验证成功的经验「结晶」成长期技能。第 5 章主角。

🎙️
学会这些名字，就是在学「指挥 AI 的词汇」

当你能说出「用 file_patch 改这一段，别整个 file_write 覆盖」，你对 AI 的指令就精确多了——这正是这门课想给你的能力。

引擎怎么知道该跑哪只「手」？

这里有一个特别优雅的小设计：靠名字对名字。大脑说工具叫 code_run，引擎就去找一个叫 do_code_run 的方法。

CODE · agent_loop.py · dispatch() 
method_name = f"do_{tool_name}"
if hasattr(self, method_name):
    ret = yield from try_call_generator(
              getattr(self, method_name), args, response)
    return ret
大白话

把工具名前面拼一个 do_。比如 code_run → do_code_run。

检查一下自己身上有没有这个名字的方法。

有的话，就去把它跑起来……

按名字精确地找到那只「手」（getattr = 按名字取出方法），把参数递给它。

跑完，把结果交回去。整个分发只用了几行。

🪄
为什么这很妙

加一件新工具，不用改循环——只要写一个 do_新名字 的方法，引擎靠命名约定就能自动找到它。这就是「少代码也能扩展」的秘诀之一。工程师管这叫约定优于配置。

9 件里最特别的一件：code_run

如果只能留一件工具，那一定是它。因为「能跑任意代码」≈「能做任意事」。

＋
缺个能力？现装。

没有处理 Excel 的库？code_run 里一句安装命令就装上了。

＋
缺个工具？现写。

需要一个抓股票的脚本？让它当场写出来、跑起来、再调试。

＋
要控制硬件 / 调外部接口？也行。

通过代码，它能驱动鼠标键盘、调摄像头、连任意 API。

🌱
这就是「能力扩展」

9 件工具看似很少，但因为有 code_run，GA 能在运行中临时长出任何新本事——再把好用的那次「结晶」成永久技能。少，但不封顶。

引擎在工具外面，悄悄加了两道规矩

光有工具还不够。引擎在「调用工具」这件事外围，加了两道隐形护栏——这也解释了你平时观察到的一些 AI 行为。

规矩一：光「贴代码」不执行？引擎会拦下来追问

有时模型会在回复里写一大段代码，却忘了真正调用工具去跑它。引擎能识破这一点：

CODE · ga.py · do_no_tool() 
blocks = re.findall(code_block_pattern, content)
if len(blocks) == 1:
    m = re.search(code_block_pattern, content)
    after_block = content[m.end():]
    if not after_block.strip():
大白话

数一数这条回复里有几个代码块。

如果恰好只有一个大代码块……

找到它的位置……

看看代码块后面还有没有别的内容……

如果后面空空如也——典型的「写完代码就停了、忘了调工具」。引擎于是反问：「这段代码你是要执行，还是只想展示给我看？」

🛑
这就是你有时看到 AI「突然反问」的原因

不是它犯傻，而是引擎在防「光说不练」——避免模型滔滔不绝写了一堆代码，结果一行都没真正跑。理解这点，你就知道该回它一句「执行它」，而不是莫名其妙。

规矩二：一轮调好几个工具？输出预算自动均分

模型一个回合里可以一次点好几个工具。但如果每个工具都返回一大坨内容，上下文瞬间就爆了。引擎的对策很简单：

CODE · ga.py · do_code_run() 
maxlen = 10000 // args.get('_tool_num', 1)
大白话

这一轮总共调了几个工具（_tool_num），就把输出额度（10000）除以几。

调 1 个工具，它独享全部额度；一次调 5 个，每个就只分到五分之一。总量恒定，绝不超支。

⚖️
还有个配套小动作

一轮里的多个工具，只有第一个会附带完整的「工作记忆」白板，后面几个直接跳过（skip），免得同样一段记忆被重复塞进去好几遍。又是一处「省 token」的精打细算——和第 4 章的主题一脉相承。

04

分层记忆：每回合擦掉白板，却从不失忆

解开第 2 章的悬念，看懂 GA「省 token、少幻觉」最核心的设计。

先回到那个悬念

第 2 章结尾我们看到这句反直觉的代码——每回合，对话被整个清空，只留一条新消息：

CODE · agent_loop.py 
# just new message, history is kept in *Session
messages = [{"role": "user",
             "content": next_prompt, ...}]
大白话

作者的注释已经剧透了答案：「历史另外存着」。

发给大脑的对话每回合都重置——不是把几十回合的聊天记录越堆越长。

那历史去哪了？它被压缩成一段精简摘要，重新贴进 next_prompt 里。

🧽
好比一块侦探白板

侦探破案不会把每天说过的每句话都钉在墙上——他擦掉白板，只留下当前最关键的线索摘要。GA 对大脑的「工作记忆」就是这么管的：每回合重写白板，而不是无限堆叠。

白板是怎么「重写」的

每回合结束，引擎会把这一步发生了什么，压成一句话摘要，追加进一条历史清单：

CODE · ga.py · turn_end_callback() 
rsumm = re.search(r"<summary>(.*?)</summary>", _c, re.DOTALL)
if rsumm: summary = rsumm.group(1).strip()
...
self.history_info.append(f'[Agent] {summary}')
大白话

在大脑的回复里，找它自己写的一段 <summary>（这一步我干了啥）。

找到了，就把这句摘要提取出来。

（找不到就用工具名兜底，并提醒大脑下次记得写摘要。）

把这句简短摘要追加到历史清单里。存的是「摘要」，不是整段原文。

下一回合开头，这份历史清单又会被「折叠」一次——太久远的部分压得更狠，只留最近 30 条原样：

CODE · ga.py · _get_anchor_prompt() 
h = self.history_info; W = 30
earlier = ... self._fold_earlier(h[:-W]) ...  # 远的折叠
h_str = "\n".join(h[-W:])         # 近 30 条原样
prompt = f"### [WORKING MEMORY]\n...<history>\n{h_str}\n</history>"
大白话

取出全部历史，定一个窗口大小 W = 30。

把 30 条以外的「远古历史」进一步折叠成更短的概要。

最近 30 条保持原样——近的清楚，远的模糊，正合人脑直觉。

拼成一段 [WORKING MEMORY]，这才是真正贴上白板、交给大脑的内容。

深入：这份「简化」，是又调了一次 LLM 吗？

很自然会以为：把长历史压成短摘要，肯定要再喊一次大脑去「总结」。其实不是。GA 刻意避开了这条路。

🗣️

常见误解

每次压缩历史，都再花一次 API 调用，让模型「读一遍旧记录、写个总结」。又慢又贵，还可能在总结时把信息搞丢、搞错。

✅

GA 的真相

那句摘要是模型在「本来就要回复的那一轮」里顺手写的；引擎只用一行正则把它抠出来。零额外调用。

CODE · ga.py · turn_end_callback() 
rsumm = re.search(r"<summary>(.*?)</summary>", _c, re.DOTALL)
if rsumm: summary = rsumm.group(1).strip()
else:
    ... summary = f"{tool_name}, args: {clean_args}"
大白话

在模型这一轮的回复里，用正则找它自己写的 <summary> 标签。

找到了，就把这段摘要抠出来——这就完事了，没有第二次 API 调用。

万一模型偷懒没写……

就直接拿「工具名 + 参数」当摘要兜底，连模型都不麻烦。

🧮
那「更狠的折叠」呢？也是纯代码

把更早的历史压成 xxx（10 turns） 这种一行概要，靠的是数数 + 字符串切片（_fold_earlier），同样不碰 LLM。确定性操作，又快又免费，还不会像 AI 总结那样偶尔「编出」原本没有的东西。

🎯
为什么这个选择很关键

如果「简化记忆」本身还要调 LLM，就和第 4 章的目标（省钱、少幻觉）自相矛盾了。GA 用「让模型搭便车自报 + 代码折叠」替代「额外 LLM 总结」，是把省钱和防幻觉贯彻到底的一步。

这就是「省 token、少幻觉」的来源

别的智能体常把几十万字的聊天记录整团塞给大脑（上下文窗口 200K–1M）。GA 全程压在 30K 以内：更便宜、噪声更少、模型更不容易被带偏。

再省一刀：读过的内容，别重复收费

还有个容易被忽略的省钱细节。每回合发给大脑的开头那一大段（系统说明书 + 记忆）几乎不变——如果每次都按全价算钱，太亏了。

🏷️
像复印店的「常用模板」

复印店会把你常印的模板存着，下次只收新增页的钱。大模型也有这功能,叫 Prompt 缓存——但你得主动给「不变的部分」贴个标记，它才会缓存。

user_idxs = [i for i, m in enumerate(messages)
             if m.get('role') == 'user']
for idx in user_idxs[-2:]:
    ... 'cache_control': {'type': 'ephemeral'}
大白话

找出对话里所有「用户消息」的位置。

（一句紧凑的列表推导式，把它们的下标都收齐。）

只给最后两条打标记——前面不变的开头就能被整段缓存命中。

贴上 cache_control: ephemeral 这个标签，告诉模型：「这段可以缓存，下次别重新收费」。

💸
小标记，大节省

命中缓存的部分,价格可能只有原价的十分之一。配合前面「上下文本就压在 30K 以内」，GA 的每一轮都既短、又大量复用缓存——省钱省到了骨子里。（仅对 Claude / Anthropic 这类支持缓存的模型生效。）

白板之外：磁盘上的「档案柜」

白板是短期记忆，任务结束就没了。真正长期沉淀的知识，存在磁盘上、分成 5 层。点标签切换看每一层：

📜
L0 · 元规则（Meta Rules）

最高宪法：核心行为准则、系统红线。比如「改自身源码先请示」「写记忆前先读 L0 核验」。永远不变。

🗂️
L1 · 洞察索引（Insight Index）

一份极简「目录」，用来快速路由：哪类任务该去翻哪份 SOP。每回合都带在身边，所以必须极小。

🗃️
L2 · 全局事实（Global Facts）

长期稳定的事实档案：常用路径、配置、凭证位置。验证过、长期有效的才进来。

📘
L3 · 任务技能 / SOP

可复用的「操作手册」：某类任务怎么一步步做、有哪些坑。比如读微信记录、抓股票的完整流程。第 5 章的产物就落在这里。

📦
L4 · 会话归档（Session Archive）

已完成任务的历史记录归档，供超长周期回忆——「上个月那次我是怎么搞定的？」

越往上越「死」（很少改动、人人遵守），越往下越「活」（不断新增、随用随长）。

🔗
短期白板 × 长期档案，两套记忆配合

干活时大脑看的是精简白板；遇到没把握的，就顺着 L1 索引去翻 L2 / L3 的档案。这套分工，让一个小上下文也能调动海量沉淀知识。

记忆什么时候被「写」进去？

关键点：GA 不是想起来才记。记忆的写入有固定时机，分两种来源——大脑主动写，和引擎按回合节奏「敲桌子」催它写。

①
任务开始 / 中途 —— 大脑主动写白板

大脑判断「这个重点后面要一直记着」，就调 update_working_checkpoint，把它钉进短期 工作记忆 的 key_info。

⑦
每满 7 回合 —— 引擎催「保存关键上下文」

怕长任务里关键信息被折叠丢掉，引擎主动提醒大脑调 update_working_checkpoint，并建议「别再无效重试，换策略 / 回看 SOP」。

㉕
每满 25 回合 —— 引擎催「写进文件」

提醒大脑把检查点、关键发现、试过的方法落到磁盘文件，避免任务太长时白板折叠把它冲掉。

💎
任务完成且有验证经验 —— 结晶进长期记忆

调 start_long_term_update，把验证成功的事实写进 L2、把流程写进 L3。这一步是第 5 章的主角。

⏰
为什么按「回合数」催，而不是凭感觉？

因为大脑干得起劲时容易「只顾往前冲、忘了存档」。引擎用 7、25 这样的固定节拍当闹钟，强制它定期把关键信息落袋——这是长任务不「半路失忆」的保险。

记忆在哪里被「读」出来用？

写只是一半。记忆真正发挥作用，是在这 4 个被读取的时机——它们决定了大脑每一步「带着多少记忆上场」。

🚀
启动时 —— 记忆被拼进「系统说明书」

程序一开机，就把 L0 行为准则 + L1 全局索引塞进系统提示词。大脑第一句话还没说，已经知道「我有哪些记忆可查」。

🦾
每个工具执行后 —— 重建白板

每跑完一件工具，_get_anchor_prompt 就把折叠后的历史 + key_info + 相关 SOP 重新拼成 [WORKING MEMORY] 贴回去。这是最高频的一次读取。

🔄
每满 10 回合 —— 重新注入 L1 索引

怕大脑聊久了「忘了自己有哪些记忆」，引擎每 10 轮把全局记忆（L1 索引 + 当前目录）整段重贴一次。

📖
卡壳 / 拿不准时 —— 主动翻档案

大脑顺着 L1 索引，用 file_read 去翻 L2 的事实、L3 的 SOP。属于「按需查阅」，不是每回合都读。

CODE · agentmain.py · get_system_prompt() 
prompt += f"\nToday: {time.strftime('%Y-%m-%d %a')}\n"
prompt += get_global_memory()
return prompt
大白话

系统说明书拼好后，先加上今天的日期（让它知道「现在是哪天」）。

再把 get_global_memory()——也就是 L1 全局索引 + 当前目录——整段拼进去。

交回这份「说明书 + 记忆」。这就是启动时记忆被读取、注入的那一下。

🔗
读和写常常接力发生

注意一个细节：当大脑用 file_read 读了一份 SOP，引擎会立刻提示它「把这份 SOP 里的关键点提取出来，写进 working memory」。一次读取，往往紧跟一次写入——记忆就这样在任务里不断流动。

把读和写，串成一条任务时间线

假设 GA 在做「盯盘选股」，整个任务 30 个回合。把记忆的读 / 写标在时间轴上，长这样：

T0
【读】启动注入

系统提示词带上 L0 + L1。大脑一上来就知道有 scheduled_task_sop 这类技能可查。

T2
【读】翻 L3 SOP

大脑 file_read 读选股 SOP → 引擎提示「提取要点」→【写】把关键步骤钉进 working memory。

T7
【写】引擎催存档

满 7 回合，引擎提醒 → 大脑调 update_working_checkpoint 保存「已装好 mootdx、筛选条件是 EXPMA 金叉」。

T10
【读】重注入 L1

满 10 回合，全局索引整段重贴一次，防止聊久了「忘了记忆库长啥样」。

T30
【写】结晶进长期记忆

任务成功，调 start_long_term_update：把「选股完整流程」写进 L3，下次一句话复用。

🫀
记忆不是「存了就放着」，而是全程在跳动

启动读一次、每步重建白板、每 10 轮重注入、按需翻档案、关键节点强制存档、完工结晶——读与写交替贯穿整个任务。这套节奏，才是「小上下文也不丢线索」的真正保障。

自我进化：越用越强的秘密

GA 把每次成功的摸索「结晶」成一个技能存进记忆——下次同类任务，一句话就能复用。

一个你身体里就有的比喻

第一次学骑自行车，你要盯着每个动作、摔好几跤。学会之后呢？跨上去就走，根本不用想——那叫肌肉记忆。

🚲
GA 也有「肌肉记忆」

第一次做一件新事，它要装库、写脚本、调试、踩坑；做成之后，它把这套流程结晶成一份技能笔记。下次同类任务，直接照做，不再从零摸索。

第一次 vs 之后每一次

同一句指令，第一次和往后的成本天差地别——这正是「自我进化」最直观的样子。

直接翻出那份技能，一句话调用。秒级完成。

「结晶」是哪件工具干的

还记得第 3 章那件特殊的记忆工具 start_long_term_update 吗？任务做完、有值得记的东西时，它就启动「结晶」流程：

CODE · ga.py · do_start_long_term_update() 
prompt = '''### [总结提炼经验] ...请提取最近一次任务中
【事实验证成功且长期有效】的环境事实、用户偏好、
重要步骤，更新记忆。
**如果没有经验证的、未来能用上的信息，忽略本次调用！**
**只能提取行动验证成功的信息** ...'''
大白话

它给大脑下一道「结算」指令：回顾刚才的任务，挑出值得长期记住的东西。

只挑三类：验证过的环境事实、用户偏好、关键步骤。

划重点：没有「验证成功、未来用得上」的信息，就直接跳过，绝不为记而记。

再划重点：只准记行动验证成功的东西——做过但没验证的，不算。

🚫
为什么对「记什么」这么严？

因为记忆会污染未来的判断。如果把没验证的猜测、临时变量、通用常识一股脑塞进去，记忆库会越来越脏，反而拖累它。只结晶被现实验证过的经验——这是保持「越用越准」而不是「越用越乱」的关键纪律。
