分层记忆：每回合擦掉白板，却从不失忆

解开第 2 章的悬念，看懂 GA「省 token、少幻觉」最核心的设计。

先回到那个悬念

第 2 章结尾我们看到这句反直觉的代码——每回合，对话被整个清空，只留一条新消息：

CODE · agent_loop.py 
# just new message, history is kept in *Session
messages = [{"role": "user",
             "content": next_prompt, ...}]
大白话

作者的注释已经剧透了答案：「历史另外存着」。

发给大脑的对话每回合都重置——不是把几十回合的聊天记录越堆越长。

那历史去哪了？它被压缩成一段精简摘要，重新贴进 next_prompt 里。

🧽
好比一块侦探白板

侦探破案不会把每天说过的每句话都钉在墙上——他擦掉白板，只留下当前最关键的线索摘要。GA 对大脑的「工作记忆」就是这么管的：每回合重写白板，而不是无限堆叠。

白板是怎么「重写」的

每回合结束，引擎会把这一步发生了什么，压成一句话摘要，追加进一条历史清单：

CODE · ga.py · turn_end_callback() 
rsumm = re.search(r"<summary>(.*?)</summary>", _c, re.DOTALL)
if rsumm: summary = rsumm.group(1).strip()
...
self.history_info.append(f'[Agent] {summary}')
大白话

在大脑的回复里，找它自己写的一段 <summary>（这一步我干了啥）。

找到了，就把这句摘要提取出来。

（找不到就用工具名兜底，并提醒大脑下次记得写摘要。）

把这句简短摘要追加到历史清单里。存的是「摘要」，不是整段原文。

下一回合开头，这份历史清单又会被「折叠」一次——太久远的部分压得更狠，只留最近 30 条原样：

CODE · ga.py · _get_anchor_prompt() 
h = self.history_info; W = 30
earlier = ... self._fold_earlier(h[:-W]) ...  # 远的折叠
h_str = "\n".join(h[-W:])         # 近 30 条原样
prompt = f"### [WORKING MEMORY]\n...<history>\n{h_str}\n</history>"
大白话

取出全部历史，定一个窗口大小 W = 30。

把 30 条以外的「远古历史」进一步折叠成更短的概要。

最近 30 条保持原样——近的清楚，远的模糊，正合人脑直觉。

拼成一段 [WORKING MEMORY]，这才是真正贴上白板、交给大脑的内容。

深入：这份「简化」，是又调了一次 LLM 吗？

很自然会以为：把长历史压成短摘要，肯定要再喊一次大脑去「总结」。其实不是。GA 刻意避开了这条路。

🗣️

常见误解

每次压缩历史，都再花一次 API 调用，让模型「读一遍旧记录、写个总结」。又慢又贵，还可能在总结时把信息搞丢、搞错。

✅

GA 的真相

那句摘要是模型在「本来就要回复的那一轮」里顺手写的；引擎只用一行正则把它抠出来。零额外调用。

CODE · ga.py · turn_end_callback() 
rsumm = re.search(r"<summary>(.*?)</summary>", _c, re.DOTALL)
if rsumm: summary = rsumm.group(1).strip()
else:
    ... summary = f"{tool_name}, args: {clean_args}"
大白话

在模型这一轮的回复里，用正则找它自己写的 <summary> 标签。

找到了，就把这段摘要抠出来——这就完事了，没有第二次 API 调用。

万一模型偷懒没写……

就直接拿「工具名 + 参数」当摘要兜底，连模型都不麻烦。

🧮
那「更狠的折叠」呢？也是纯代码

把更早的历史压成 xxx（10 turns） 这种一行概要，靠的是数数 + 字符串切片（_fold_earlier），同样不碰 LLM。确定性操作，又快又免费，还不会像 AI 总结那样偶尔「编出」原本没有的东西。

🎯
为什么这个选择很关键

如果「简化记忆」本身还要调 LLM，就和第 4 章的目标（省钱、少幻觉）自相矛盾了。GA 用「让模型搭便车自报 + 代码折叠」替代「额外 LLM 总结」，是把省钱和防幻觉贯彻到底的一步。

💰
这就是「省 token、少幻觉」的来源

别的智能体常把几十万字的聊天记录整团塞给大脑（上下文窗口 200K–1M）。GA 全程压在 30K 以内：更便宜、噪声更少、模型更不容易被带偏。

再省一刀：读过的内容，别重复收费

还有个容易被忽略的省钱细节。每回合发给大脑的开头那一大段（系统说明书 + 记忆）几乎不变——如果每次都按全价算钱，太亏了。

🏷️
像复印店的「常用模板」

复印店会把你常印的模板存着，下次只收新增页的钱。大模型也有这功能,叫 Prompt 缓存——但你得主动给「不变的部分」贴个标记，它才会缓存。

CODE · llmcore.py · _stamp_oai_cache_markers() 
user_idxs = [i for i, m in enumerate(messages)
             if m.get('role') == 'user']
for idx in user_idxs[-2:]:
    ... 'cache_control': {'type': 'ephemeral'}
大白话

找出对话里所有「用户消息」的位置。

（一句紧凑的列表推导式，把它们的下标都收齐。）

只给最后两条打标记——前面不变的开头就能被整段缓存命中。

贴上 cache_control: ephemeral 这个标签，告诉模型：「这段可以缓存，下次别重新收费」。

💸
小标记，大节省

命中缓存的部分,价格可能只有原价的十分之一。配合前面「上下文本就压在 30K 以内」，GA 的每一轮都既短、又大量复用缓存——省钱省到了骨子里。（仅对 Claude / Anthropic 这类支持缓存的模型生效。）

白板之外：磁盘上的「档案柜」

白板是短期记忆，任务结束就没了。真正长期沉淀的知识，存在磁盘上、分成 5 层。点标签切换看每一层：

📜
L0 · 元规则（Meta Rules）

最高宪法：核心行为准则、系统红线。比如「改自身源码先请示」「写记忆前先读 L0 核验」。永远不变。

🗂️
L1 · 洞察索引（Insight Index）

一份极简「目录」，用来快速路由：哪类任务该去翻哪份 SOP。每回合都带在身边，所以必须极小。

🗃️
L2 · 全局事实（Global Facts）

长期稳定的事实档案：常用路径、配置、凭证位置。验证过、长期有效的才进来。

📘
L3 · 任务技能 / SOP

可复用的「操作手册」：某类任务怎么一步步做、有哪些坑。比如读微信记录、抓股票的完整流程。第 5 章的产物就落在这里。

📦
L4 · 会话归档（Session Archive）

已完成任务的历史记录归档，供超长周期回忆——「上个月那次我是怎么搞定的？」

越往上越「死」（很少改动、人人遵守），越往下越「活」（不断新增、随用随长）。

🔗
短期白板 × 长期档案，两套记忆配合

干活时大脑看的是精简白板；遇到没把握的，就顺着 L1 索引去翻 L2 / L3 的档案。这套分工，让一个小上下文也能调动海量沉淀知识。

记忆什么时候被「写」进去？

关键点：GA 不是想起来才记。记忆的写入有固定时机，分两种来源——大脑主动写，和引擎按回合节奏「敲桌子」催它写。

①
任务开始 / 中途 —— 大脑主动写白板

大脑判断「这个重点后面要一直记着」，就调 update_working_checkpoint，把它钉进短期 工作记忆 的 key_info。

⑦
每满 7 回合 —— 引擎催「保存关键上下文」

怕长任务里关键信息被折叠丢掉，引擎主动提醒大脑调 update_working_checkpoint，并建议「别再无效重试，换策略 / 回看 SOP」。

㉕
每满 25 回合 —— 引擎催「写进文件」

提醒大脑把检查点、关键发现、试过的方法落到磁盘文件，避免任务太长时白板折叠把它冲掉。

💎
任务完成且有验证经验 —— 结晶进长期记忆

调 start_long_term_update，把验证成功的事实写进 L2、把流程写进 L3。这一步是第 5 章的主角。

⏰
为什么按「回合数」催，而不是凭感觉？

因为大脑干得起劲时容易「只顾往前冲、忘了存档」。引擎用 7、25 这样的固定节拍当闹钟，强制它定期把关键信息落袋——这是长任务不「半路失忆」的保险。

记忆在哪里被「读」出来用？

写只是一半。记忆真正发挥作用，是在这 4 个被读取的时机——它们决定了大脑每一步「带着多少记忆上场」。

🚀
启动时 —— 记忆被拼进「系统说明书」

程序一开机，就把 L0 行为准则 + L1 全局索引塞进系统提示词。大脑第一句话还没说，已经知道「我有哪些记忆可查」。

🦾
每个工具执行后 —— 重建白板

每跑完一件工具，_get_anchor_prompt 就把折叠后的历史 + key_info + 相关 SOP 重新拼成 [WORKING MEMORY] 贴回去。这是最高频的一次读取。

🔄
每满 10 回合 —— 重新注入 L1 索引

怕大脑聊久了「忘了自己有哪些记忆」，引擎每 10 轮把全局记忆（L1 索引 + 当前目录）整段重贴一次。

📖
卡壳 / 拿不准时 —— 主动翻档案

大脑顺着 L1 索引，用 file_read 去翻 L2 的事实、L3 的 SOP。属于「按需查阅」，不是每回合都读。

CODE · agentmain.py · get_system_prompt() 
prompt += f"\nToday: {time.strftime('%Y-%m-%d %a')}\n"
prompt += get_global_memory()
return prompt
大白话

系统说明书拼好后，先加上今天的日期（让它知道「现在是哪天」）。

再把 get_global_memory()——也就是 L1 全局索引 + 当前目录——整段拼进去。

交回这份「说明书 + 记忆」。这就是启动时记忆被读取、注入的那一下。

🔗
读和写常常接力发生

注意一个细节：当大脑用 file_read 读了一份 SOP，引擎会立刻提示它「把这份 SOP 里的关键点提取出来，写进 working memory」。一次读取，往往紧跟一次写入——记忆就这样在任务里不断流动。

把读和写，串成一条任务时间线

假设 GA 在做「盯盘选股」，整个任务 30 个回合。把记忆的读 / 写标在时间轴上，长这样：

T0
【读】启动注入

系统提示词带上 L0 + L1。大脑一上来就知道有 scheduled_task_sop 这类技能可查。

T2
【读】翻 L3 SOP

大脑 file_read 读选股 SOP → 引擎提示「提取要点」→【写】把关键步骤钉进 working memory。

T7
【写】引擎催存档

满 7 回合，引擎提醒 → 大脑调 update_working_checkpoint 保存「已装好 mootdx、筛选条件是 EXPMA 金叉」。

T10
【读】重注入 L1

满 10 回合，全局索引整段重贴一次，防止聊久了「忘了记忆库长啥样」。

T30
【写】结晶进长期记忆

任务成功，调 start_long_term_update：把「选股完整流程」写进 L3，下次一句话复用。

🫀
记忆不是「存了就放着」，而是全程在跳动

启动读一次、每步重建白板、每 10 轮重注入、按需翻档案、关键节点强制存档、完工结晶——读与写交替贯穿整个任务。这套节奏，才是「小上下文也不丢线索」的真正保障。

小测验：拿它来排查问题

理解了记忆设计，你就能解释很多「为什么 AGI 智能体会这样」的现象。

有人问你：「为什么 GenericAgent 比别的智能体更省钱、更少胡说八道？」最准确的回答是？

GA 第一次成功读取了微信聊天记录，过程中踩了好几个坑。它想把「这类任务该怎么做」永久存下来，应该写进哪一层？

一个几十回合的长任务，你发现 GA 到后期好像「忘了」最初设定的筛选条件。基于记忆机制，最可能的原因 + 最该让它做什么？
