100 行的心跳

把整个智能体引擎想成一局「回合制游戏」——每一回合，AI 只做一件事。

一个好用的比喻：回合制游戏

象棋、回合制 RPG，都是「轮到你 → 你走一步 → 看棋盘变化 → 再轮到你」。GenericAgent 的引擎一模一样。

🧠
轮到 AI 行动

把当前对话发给大脑，让它想：「这一步我要调用哪个工具？」

🦾
执行这一步

引擎把大脑点名的工具真正跑一遍，拿到结果。

🔄
把棋盘变化告诉它

结果被打包成下一回合的输入，再丢回大脑。回合数 +1。

⏱️
一个「回合」= 一次工具调用

工程师把这一圈叫做一个 turn（回合）。下面的代码里，turn 这个计数器就是游戏的回合数。

心跳的第一拍：开口问大脑

每个回合的开头，引擎做的第一件事，是把整份对话发给大脑、等它回话。

CODE · agent_loop.py 
while turn < handler.max_turns:
    turn += 1
    response_gen = client.chat(messages=messages, tools=tools_schema)
    response = yield from response_gen
大白话

只要回合数还没到上限，就继续转这个圈。

回合数加 1——新的一回合开始了。

把整份对话 + 可用工具清单，发给大脑，请它思考并回话。

等大脑把话说完，收下它的完整回复 response。

📡
为什么把工具清单一起发？

大脑不会凭空知道自己有哪些「手」。每回合都把 tools_schema（工具说明书）一起递过去，它才知道这回合能选哪些工具来用。

第二拍：大脑点名，引擎动手

大脑的回话里，藏着它这一步想调用的工具。引擎要把这些「点名」一个个解析出来，然后真正去执行。

CODE · agent_loop.py 
tool_calls = [{'tool_name': tc.function.name,
               'args': json.loads(tc.function.arguments),
               'id': tc.id}
              for tc in response.tool_calls]
大白话

从大脑的回复里，把每一个「工具调用」拆出来。

记下工具的名字（比如 code_run）。

记下要传给它的参数。JSON 文本在这里被翻译回程序能用的形式。

还有一个 id，方便稍后把「结果」对回到「是哪次调用」。

大脑这回合可能一次点好几个工具，所以用列表把它们都收齐。

把一个回合，听成一段对话

如果给引擎、大脑、工具三个角色配上对话框，一个回合大概是这样聊的：

⚙️
引擎 (Loop)

第 7 回合。这是目前的对话和你能用的工具，想想下一步该干嘛？

🧠
大脑 (LLM)

我决定调用 code_run，跑这段 Python 把价格抓下来。

⚙️
引擎 (Loop)

收到，我去找名叫 do_code_run 的执行器，把它跑了。

🦾
工具 (code_run)

跑完了，结果：{"price": 18}。给你一个 StepOutcome。

⚙️
引擎 (Loop)

它没说「该退出」，那就把结果打包成下一回合的输入。第 8 回合，开始！

?

第三拍：这回合的「裁决」

每个工具跑完，都会交回一张小纸条，叫 StepOutcome。引擎只看上面三栏，就知道这局怎么继续。

CODE · agent_loop.py 
@dataclass
class StepOutcome:
    data: Any
    next_prompt: Optional[str] = None
    should_exit: bool = False
大白话

定义一张标准化的「结果纸条」。每个工具跑完都填一张。

data：这一步实际拿到的东西（比如抓到的价格）。

next_prompt：下一回合要对大脑说的话。如果留空，引擎就认为「任务做完了」，停。

should_exit：一个 真假开关。比如工具要找用户确认，就把它设成「真」，主动跳出循环等人。

🧭
三栏纸条，就是整局游戏的方向盘

「有没有下一句话」决定继续还是收工；「要不要退出」决定是否暂停等人。一个极简的约定，控制了整个循环的走向——这就是少代码却不乱的诀窍。

心跳的最后一拍：只带「新消息」进下一回合

每回合结束，引擎都会做一件乍看很怪的事——把对话列表清空，只留下这一条新消息。

CODE · agent_loop.py 
# just new message, history is kept in *Session
messages = [{"role": "user",
             "content": next_prompt,
             "tool_results": tool_results}]
大白话

代码作者亲手写的注释：「只放新消息，历史另外存着」。

把 messages 整个重置，里头只剩一条新的「用户消息」。

内容是上一步的裁决（next_prompt）……

外加这回合工具跑出来的结果。然后下一回合就从这里继续。

🤔
等等——那之前的对话不就丢了吗？

这正是 GenericAgent 最聪明、也最反直觉的设计，直接关系到它「省 token、少出错」的核心优势。我们专门留到 第 4 章 拆它。先记住这个疑问。
